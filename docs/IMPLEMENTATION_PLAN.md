# План реализации Monopoly SDS

## 1. MVP-границы (2-3 недели)
- Авторизация игроков и админов.
- Игровое поле с клетками "приз/штраф/нейтрально".
- Бросок кубика с серверной проверкой.
- Баланс игрока и применение эффектов клетки.
- Админка: CRUD поля, клеток, призов, участников.

## 2. Этапы разработки

### Этап 0: Проектирование
- Утвердить правила игры (кол-во клеток, лимит ходов, cooldown).
- Утвердить модель начисления баллов за продажи.
- Описать event-flow (roll -> move -> apply cell effect -> save transaction).

### Этап 1: Backend Foundation
- Поднять FastAPI приложение (`backend/app/main.py`).
- Настроить БД, миграции, базовые таблицы.
- Реализовать auth (JWT access/refresh).
- Реализовать роли и права доступа.

### Этап 2: Admin API
- `POST/PUT/DELETE /admin/board` - управление полем.
- `POST/PUT/DELETE /admin/rewards` - управление призами.
- `GET /admin/participants` - трекинг игроков.
- `GET /admin/transactions` - аудит операций.

### Этап 3: Game API
- `GET /game/state` - состояние игрока (позиция, баланс, доступные действия).
- `POST /game/roll` - бросок кубика.
- `POST /game/reward/redeem` - покупка приза.
- `GET /game/history` - история ходов/начислений.

### Этап 4: Frontend Player
- Страницы: логин/регистрация, игровое поле, профиль/история.
- Визуал поля, анимация хода, модалки события клетки.
- Экран покупки приза и отображение баланса.

### Этап 5: Frontend Admin
- Страница конструктора поля (drag/drop опционально).
- Страница призов/штрафов.
- Дашборд участников и активности.

### Этап 6: Интеграция продаж
- Внешний endpoint `POST /sales/events` для начисления баллов.
- Идемпотентность по `external_event_id`.
- Логи ошибок и ретраи.

### Этап 7: Тесты и релиз
- Unit-тесты движка игры и баланса.
- Integration-тесты API.
- E2E критических сценариев.
- Docker Compose для локального запуска.

## 3. KPI MVP
- % зарегистрированных игроков, совершивших хотя бы 1 ход.
- Среднее число ходов на игрока.
- Конверсия накопление -> обмен на приз.
- Ошибки API < 1%.

## 4. Риски
- Фрод бросков кубика -> только серверный random + лимиты.
- Нечеткая бизнес-логика призов -> конфигурируемые правила в админке.
- Сложность интерфейса поля -> ранний UX-прототип и юзабилити-тесты.
